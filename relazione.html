Introduzione
Obiettivo · Contesto · Problema

Obiettivo del progetto

Il progetto consiste nella realizzazione di un sito web statico dedicato al Campionato Mondiale di Motociclismo 2026 (Motomondiale). L'obiettivo principale è offrire un'unica piattaforma informativa che raccolga classifiche, risultati, schede piloti e notizie per le tre categorie — MotoGP, Moto2 e Moto3 — in un'interfaccia moderna, responsiva e consultabile anche offline.

Contesto

Il sito nasce come applicazione web puramente client-side (front-end), senza necessità di un server back-end o di un database remoto. Tutti i dati sono incorporati direttamente nel codice JavaScript dei file HTML, una scelta che rende il progetto distribuibile semplicemente copiando la cartella su qualsiasi hosting statico.

Problema da risolvere

I siti ufficiali del Motomondiale sono spesso lenti, appesantiti da pubblicità e difficili da navigare su mobile. Il progetto vuole offrire un'alternativa leggera, veloce e accessibile che metta al centro l'informazione tecnica: punti, risultati sessione per sessione, palmares dei piloti e storico dell'albo d'oro dal 1949.

Analisi dei Requisiti
Funzionali · Non funzionali · HW/SW

Analisi applicativa

Il dominio applicativo è quello dell'informazione sportiva. Gli utenti sono appassionati di motociclismo che vogliono consultare rapidamente classifiche, risultati e biografie dei piloti. L'interazione avviene tramite browser su desktop, tablet e smartphone. Non è richiesta autenticazione né la creazione di account.

Requisiti hardware e software

Client
Qualsiasi device
PC, Mac, smartphone, tablet. Browser moderno (Chrome, Firefox, Safari, Edge).
Server
Hosting statico
Netlify, GitHub Pages, Vercel o qualsiasi server HTTP. Nessun PHP/Python/Node richiesto.
Connessione
Opzionale (PWA)
Grazie al Service Worker, il sito funziona anche offline dopo la prima visita.

Requisiti funzionali

  • Il sistema deve permettere la visualizzazione delle classifiche piloti e costruttori per MotoGP, Moto2 e Moto3, aggiornate dopo ogni round.
  • Il sistema deve permettere la consultazione dei risultati sessione per sessione (FP, Qualifiche Q1/Q2, Sprint, Gara) per ogni round disputato.
  • Il sistema deve permettere di filtrare la classifica per costruttore e confrontare due piloti della stessa categoria.
  • Il sistema deve permettere di consultare la scheda completa di ogni pilota con palmares, storico stagioni, grafico punti per round e piloti vicini in classifica.
  • Il sistema deve permettere la lettura di notizie delle tre categorie con apertura in-page dell'articolo completo.
  • Il sistema deve permettere la ricerca di un pilota per nome in tutte le pagine tramite la scorciatoia da tastiera /.

Requisiti non funzionali

Prestazioni: caricamento iniziale sotto 2 secondi su connessione 4G. Nessuna richiesta di rete obbligatoria per la navigazione (tutti i dati sono embedded).
Responsività: layout adattivo per schermi da 320px a 2560px. Breakpoint principale a 1100px (hamburger) e 768px (colonne mobili).
Usabilità: navigazione raggiungibile con al massimo 2 click da qualsiasi pagina. Tema scuro/chiaro selezionabile con preferenza salvata.
Offline: il Service Worker (PWA) mette in cache tutte le pagine principali alla prima visita, consentendo la consultazione senza connessione.

Progettazione
Architettura · Struttura · Dati

Architettura generale

Il sito segue un'architettura Single-File Component senza framework: ogni pagina HTML contiene al proprio interno il CSS specifico (nel <style>), il markup HTML semantico e il codice JavaScript (nei tag <script>). Questo approccio elimina le dipendenze esterne e rende il sito completamente autonomo.

HTML5 → struttura semantica
CSS3 → variabili custom, flexbox, grid, media query
JavaScript ES2022 → render dinamico, dati embedded, DOM manipulation
Fetch API → dati live da MotoGP PulseLive (fallback su dati statici)
Service Worker → cache offline, installazione come app (PWA)
localStorage → tema, round selezionato, stato toast

Struttura delle pagine

  • index.html — Home: notizie, countdown prossimo GP, barra progressione stagione, top-5 classifiche, calendario.
  • motogp/moto2/moto3.html — Pagine categoria: notizie filtrate, classifica parziale, entry list per team.
  • piloti.html — Lista completa 79 piloti con ricerca, filtro categoria, corona 👑 sul leader.
  • pilota.html — Scheda singolo pilota: palmares, tabella stagioni, grafico punti per round, piloti simili.
  • classifica.html — Classifiche complete piloti + costruttori, layout 30/70, filtro costruttore, confronto piloti.
  • risultati.html — Risultati sessione per sessione (FP1/FP2/Q1/Q2/Sprint/Gara) per i primi 3 round.
  • calendario.html — 22 round stagionali con popup per i round futuri.
  • stagione.html — Riepilogo statistico stagionale con sezione "piloti in forma".
  • albo_doro.html — Albo d'oro dal 1949 con sidebar ricerca su tutte le categorie.

Struttura dei dati JavaScript

I dati sono organizzati in oggetti JavaScript costanti, definiti in cima ai file che li utilizzano:

Struttura della Navigazione
Header · Mobile Menu · Routing

Sistema di navigazione multi-breakpoint

La navigazione è gestita interamente in CSS e JavaScript vanilla. L'header è sticky (rimane fisso durante lo scroll) con z-index 200 per restare sopra a tutti gli altri elementi. Contiene tre componenti: logo, nav desktop e div header-right con toggle tema e hamburger.

/* Breakpoint principale: 1100px */ @media (max-width: 1100px) { nav { display: none !important; } .hamburger { display: flex !important; } } @media (min-width: 1101px) { .hamburger { display: none !important; } }

Il breakpoint è stato fissato a 1100px invece del classico 768px perché il nav contiene 12 link — a 1100px i link iniziano a essere troppo compressi per essere leggibili. Sotto quella soglia appare il menu hamburger che apre un dropdown a piena larghezza con tutti i link in colonna.

Routing client-side

Non essendoci un server che gestisce le route, la navigazione tra pagine avviene tramite link HTML standard con parametri URL. La pagina pilota.html riceve l'identificativo del pilota tramite query string (?id=marco_bezzecchi) e lo legge con URLSearchParams:

const riderId = new URLSearchParams(location.search).get('id'); const data = RIDERS[riderId]; // lookup O(1) nell'oggetto

Gli slug dei piloti (es. marc_marquez) vengono calcolati al volo dal nome con la funzione nameToSlug() che normalizza gli accenti, rimuove i caratteri speciali e sostituisce gli spazi con underscore — garantendo consistenza tra la classifica e la scheda pilota.

Gestione del tema scuro/chiaro

Il tema è implementato tramite l'attributo HTML data-theme="dark|light" sull'elemento radice <html>. Tutte le pagine usano lo stesso set di CSS Custom Properties (--dark, --mid, --card, --border, --muted, --white) che vengono ridefinite nel selettore [data-theme="light"]. Questo permette di cambiare l'intero tema modificando solo 8 variabili senza JavaScript che tocchi i singoli elementi.

5b
Performance e Ottimizzazione
Caricamento · Immagini · Cache

Nessuna dipendenza JavaScript esterna

La scelta di non usare framework (React, Vue, Angular) o librerie (jQuery, Bootstrap) elimina completamente il problema del "bundle size". Un progetto React minimo scarica ~150KB di JavaScript prima ancora di mostrare un pixel; questo sito scarica solo il file HTML della pagina richiesta — tipicamente 80-200KB, inclusi tutti i dati.

Ottimizzazione delle immagini

  • Formato WebP — le 79 foto piloti sono in formato WebP, che offre una compressione mediamente il 30% migliore rispetto a JPEG mantenendo la stessa qualità visiva.
  • Lazy loading — le immagini fuori dalla viewport iniziale vengono caricate solo quando necessario grazie all'attributo loading="lazy".
  • Dimensioni ottimizzate — le 22 immagini dei tracciati sono in PNG con risoluzione adeguata al contenitore (non più grande del necessario).

CSS Custom Properties e variabili

L'uso di CSS Custom Properties permette al browser di calcolare i valori una sola volta e riutilizzarli in tutti gli elementi che li referenziano. Cambiare il tema richiede una singola operazione DOM (setAttribute('data-theme', 'light')) invece di migliaia di modifiche inline ai singoli elementi.

Tipografia e risorse esterne

  • Bebas Neue (Google Fonts) — font display per titoli, numeri e label. Scelto per il carattere sport/racing.
  • Barlow Condensed (Google Fonts) — font compresso per la navigazione e le label secondarie. Ottima leggibilità a dimensioni ridotte.
  • Barlow (Google Fonts) — font di testo per i paragrafi e il corpo degli articoli.
  • Immagini — 79 foto piloti in formato WebP (ottimizzato per il web) e 22 immagini dei tracciati in PNG.

Motivazioni delle scelte

La scelta di Vanilla JS senza framework (React, Vue, ecc.) è stata deliberata: per un sito di contenuti statici aggiunge complessità inutile, aumenta il bundle e riduce le performance. Con HTML/CSS/JS puri il sito carica in meno di 1 secondo e non dipende da versioni di librerie esterne che potrebbero diventare obsolete.

L'assenza di un back-end permette di ospitare il sito gratuitamente su Netlify, GitHub Pages o Vercel — servizi di hosting statico che offrono CDN globale, HTTPS automatico e deploy in 30 secondi.

L'unica chiamata di rete del sito è la Fetch API verso l'endpoint PulseLive per arricchire la scheda pilota con dati live. Tutti gli altri dati — classifiche, risultati, notizie, palmares — sono incorporati nel codice e disponibili offline.

Implementazione
Funzionalità chiave · Codice · Pattern

Pattern architetturale: render dinamico dal JS

Le tabelle classifiche, le schede piloti e i risultati gara non sono scritti staticamente nell'HTML ma generati da funzioni JavaScript che leggono i dati dagli oggetti DATA, RIDERS e RESULTS e costruiscono il markup con template literals:

// classifica.html — render riga classifica function renderRow(r, i, maxPts) { const barW = Math.round((r.pts / maxPts) * 100); return `<tr data-slug="${nameToSlug(r.name)}"> <td><span class="pos-num">${i+1}</span></td> <td>${r.flag}</td> <td> <div class="rider-name">${r.name}</div> <div class="rider-team">${r.team}</div> </td> <td>${r.pts}</td> <td><div class="pts-bar" style="width:${barW}%"></div></td> </tr>`; }

Confronto piloti

La funzione compareRiders() in classifica.html genera in tempo reale una card comparativa con barra visiva proporzionale ai punti, evidenziazione del "vincitore" per ogni metrica e gestione del caso in cui i valori siano molto sbilanciati (es. 81 vs 1 — usa Math.max(w, 8) per la larghezza minima della barra).

Calcolo costruttori

La classifica costruttori usa i punteggi ufficiali FIM: per ogni round si prende il miglior risultato singolo per costruttore, non la somma di tutti i piloti. Per la MotoGP viene conteggiata anche la Sprint Race separatamente. I dati statici finali sono hardcodati dopo il calcolo per garantire accuratezza (es. Aprilia 101 pt dopo 3 round).

Service Worker — strategia di cache

// sw.js — Network-first, fallback su cache self.addEventListener('fetch', e => { e.respondWith( caches.match(e.request).then(cached => { if (cached) return cached; return fetch(e.request).then(resp => { caches.open(CACHE).then(c => c.put(e.request, resp.clone())); return resp; }).catch(() => caches.match('index.html')); }) ); });

Responsività — breakpoint system

  • > 1100px: nav desktop completo con tutti gli 11 link.
  • 768px–1100px: hamburger menu, layout a colonna singola per sezioni a due colonne.
  • < 768px: font scalati, padding ridotti, tabelle con colonne nascoste (bar-cell, gap-cell).
  • < 500px: grid piloti 2 colonne, hero font con clamp().

Dark / Light mode

Il tema è gestito tramite l'attributo data-theme sull'elemento <html> e un set di CSS custom properties ridefinite nel selettore [data-theme="light"]. La preferenza viene salvata in localStorage e ripristinata al caricamento da initTheme().

Funzionalità UX implementate

Nel corso dello sviluppo sono state aggiunte diverse funzionalità di esperienza utente:

  • Scorciatoie da tastiera — tasti M, 2, 3, C, R, P, H e / con toast di conferma prima della navigazione.
  • Bottone "Torna indietro" dinamico — implementato con sessionStorage: ogni click su un link interno salva la pagina corrente; alla destinazione compare "← Torna a [origine]".
  • Animazioni fade-in scroll — IntersectionObserver con stagger di 60ms applicato a card, sezioni e stat-box. Zero librerie esterne.
  • Loading skeleton in piloti.html — 12 card con animazione shimmer visibili durante il caricamento, rimosse automaticamente.
  • Service Worker v3 — strategia stale-while-revalidate: risponde dalla cache istantaneamente mentre aggiorna in background.
  • Favicon SVG — icona scalabile con "M26" rosso su sfondo nero, compatibile con tutti i browser moderni.

Accessibilità e Usabilità
ARIA · Keyboard · UX

Accessibilità da tastiera

Tutta la navigazione del sito è raggiungibile da tastiera senza mouse. La scorciatoia / apre la lista piloti da qualsiasi pagina. I bottoni interattivi hanno attributi aria-label per i lettori di schermo. Il toggle tema ha aria-label="Cambia tema".

Attributi ARIA e semantica HTML

  • Tag semantici: <header>, <nav>, <main>, <section>, <article>, <footer> per strutturare correttamente il documento.
  • Attributo aria-label su hamburger, theme-toggle e scroll-top-btn.
  • Attributo alt su tutte le immagini dei piloti per i lettori di schermo.
  • Attributo lang="it" sull'elemento <html> per indicare la lingua al browser.

Principi UX applicati

Feedback visivo immediato: ogni elemento interattivo (link, bottone, riga tabella) cambia aspetto al hover con transizioni CSS di 150ms — abbastanza veloci da sembrare istantanee, ma percettibili come risposta all'azione.

Gerarchia visiva: ogni pagina ha un unico punto di focus principale (hero, titolo categoria, nome pilota) identificato da font size e peso tipografico superiori. Le informazioni secondarie usano colori più chiari (var(--muted)) per non competere con il contenuto principale.

Consistenza: ogni pagina usa gli stessi componenti (header, footer, section-title, card) con gli stessi stili. L'utente impara il sistema una volta sola e può navigare qualsiasi pagina senza sorprese.

Indicatori di stato

  • Round disputati vs futuri: nel calendario, i round completati portano ai risultati, quello successivo pulsa in giallo, i futuri mostrano un popup informativo invece di navigare a una pagina vuota.
  • Leader di classifica: il pilota con più punti di ogni categoria ha la corona 👑 sulla card in piloti.html.
  • API live: nella scheda pilota, se la chiamata all'API MotoGP riesce, appare il badge "✓ Live API" in verde nella sidebar.

Sviluppi Futuri
Miglioramenti · Scalabilità

Back-end e dati in tempo reale

L'evoluzione naturale del progetto prevede l'introduzione di un back-end leggero (Node.js + Express o Python Flask) che interroga le API ufficiali MotoGP dopo ogni sessione e aggiorna automaticamente un file JSON centralizzato. Questo eliminerebbe la necessità di aggiornamento manuale dei dati.

Database centralizzato

Sostituire i dati embedded nei file HTML con un unico file database.json letto via Fetch da tutte le pagine. Ridurrebbe drasticamente la dimensione dei file HTML (da ~18.000 a ~3.000 righe) e renderebbe gli aggiornamenti istantanei modificando un solo file.

Notifiche push

Tramite le Push API del browser (già supportata dai Service Worker PWA), si potrebbe notificare l'utente al termine di ogni sessione con il vincitore e i punti — senza app nativa.

Applicazione nativa

Il sito è già installabile come PWA (Progressive Web App) su Android e iOS. Per una vera app nativa su App Store/Google Play si potrebbe usare Capacitor (Ionic) che avvolge il codice HTML/JS esistente in un container nativo, richiede circa 1-2 giorni di configurazione aggiuntiva.

Scalabilità

  • Aggiunta di nuovi round: basta estendere l'oggetto RESULTS con i dati del round successivo.
  • Supporto stagioni precedenti: una pagina archivio.html con filtro per anno potrebbe mostrare le classifiche storiche.
  • Internazionalizzazione: i file di testo sono già strutturati in modo da supportare una traduzione in inglese o spagnolo con un oggetto i18n.
07
Conclusioni
Risultati · Valutazione · Criticità

Il progetto ha raggiunto tutti gli obiettivi iniziali: un sito informativo completo sul Motomondiale 2026, navigabile da qualsiasi dispositivo, con dati aggiornati ai primi tre round stagionali e funzionante anche offline grazie alla tecnologia PWA.

Risultati ottenuti

  • 12 pagine funzionanti con navigazione coerente e tema scuro/chiaro.
  • 79 piloti con scheda completa, palmares e grafico punti per round.
  • Classifiche dinamiche con confronto piloti e filtro costruttore.
  • Risultati completi per 3 round (Thailandia, Brasile, USA) con tutte le sessioni.
  • Installabile come app su Android, iOS e desktop.

Criticità incontrate e soluzioni

CSS con definizioni duplicate — L'accumulo di sessioni di sviluppo aveva prodotto ridefinizioni CSS conflittuali dell'header. Risolto usando !important su un blocco unico di override aggiunto in coda al <style>.

MutationObserver in loop — L'aggiunta delle medaglie 🥇🥈🥉 tramite un Observer che osservava il nodo che modificava ha generato un loop infinito bloccando la pagina risultati. Risolto rimuovendo l'Observer e chiamando la funzione direttamente dopo ogni render.

Dati costruttori — Il calcolo automatico dei punti costruttori non corrispondeva ai valori ufficiali per via dei punteggi dimezzati al round 1 Moto2 (bandiera rossa) e delle regole FIM. Risolto con punteggi statici hardcodati post-calcolo.

Valutazione complessiva

Il progetto dimostra come sia possibile realizzare un'applicazione web complessa, performante e responsive usando esclusivamente le tecnologie base del web (HTML, CSS, JavaScript) senza dipendenze esterne, mantenendo il codice comprensibile, manutenibile e aggiornabile.
